## 🧪 Phase 1: Pre-Upload QA Testing

Stake expects games submitted by vendors to be flawless. Before touching the Stake Admin Control Panel (ACP), you must perform a strict suite of QA tests locally.

### How and Where to Test Locally
1. **Launch the Production Bundle:**
   Instead of testing in development mode (`npm run dev`), you must test the optimized production build.
   ```bash
   npm run build
   npx vite preview
   ```
   *Open `http://localhost:4173` in an Incognito window.*

### Mandatory Test Scenarios (The QA Checklist)
- [ ] **The "Throttle" Test:** Open Chrome DevTools > Network tab. Change throttling to "Fast 3G" or "Slow 3G". Verify that the loading spinner doesn't crash and the reels wait patiently for the API simulated delay.
- [ ] **The "Orientation Stress" Test:** Using Chrome DevTools "Device Toolbar", select an iPhone 14. Rapidly rotate the screen between Portrait and Landscape mid-spin. Ensure the canvas resizes seamlessly without stretching the candies.
- [ ] **The "Rage Click" Check:** Tap the 🔁 Spin button 10 times per second. Verify that the `options.checkClick` lock functions—only one bet should be deducted, and only one API call should traverse the network.
- [ ] **The "Amnesia" Protocol:** Start a spin, and the moment the grid begins cascading, close the browser tab completely. Re-open the game. The `localStorage` system must immediately pop up a window showing the result of that interrupted spin.
- [ ] **The iOS WebAudio Rule:** Open Safari (if available) or an iPhone emulator. Ensure no audio plays until you physically click the "PLAY" button on the boot screen.

## ⚙️ Phase 2: Code Adjustments Required Before Upload

Currently, the `StakeEngineClient.ts` is operating in an encapsulated "Mock / Demo" state designed for local development. **When you are ready to upload the game to Stake's live staging servers**, you must update a few configurations:

### 1. Update the RGS URL Endpoint
In `/src/engine/StakeEngineClient.ts`, locate the `rgsUrl` constructor assignment.
- **Change:** `this.rgsUrl = new URLSearchParams(window.location.search).get('rgs_url') || 'mock://local';`
- **To:** The actual API Gateway endpoint provided by the Stake integration team (e.g., `https://api.stake.com/providers/v1/your-username`).
- *Note:* In production, the game fetches `sessionID` strings from the browser URL automatically (e.g., `?sessionID=xxx`). Ensure this parsing remains untouched.

### 2. Verify Asset Paths (Base URL)
If Stake requires you to host the frontend assets (images, music) on a specific CloudFront CDN, you must update your `vite.config.ts`.
- Set the `base: '/'` property to match Stake's CDN bucket path (e.g., `base: 'https://cdn.stake.com/games/sweet-cluster-1000/'`). If Stake hosts the `.zip` file natively on their root domain, leave it as `./` or `/`.

## 📤 Phase 3: The Stake Upload Process (ACP)

Stake manages thousands of vendor slots via an Admin Control Panel (ACP). Here is the exact process to submit your game.

### Step A: Prepare the Artifacts
You will need two files ready on your desktop:
1. `sweet_cluster_1000.zip` (Generated by running `npm run bundle` in your terminal).
2. `math-engine/output/game_config.json` (The mathematical rulebook).

### Step B: The Upload Sequence
1. Log into the **Stake Developer Portal / Vendor ACP** using your client credentials.
2. Navigate to **Game Management -> Portfolio -> Create New Game**.
3. **Basic Details:**
   - **Name:** Sweet Cluster 1000
   - **Type:** Slot
   - **Grid Size:** 7x7
   - **Max Win:** 25,000x
   - **RTP:** 95.30%
4. **Asset Upload (Frontend):**
   - Look for the "Client Bundle" or "Static Assets" drag-and-drop box.
   - Upload `sweet_cluster_1000.zip`. The ACP will automatically unzip and map the `index.html` as the game's launch point.
5. **RGS Config Upload (Backend):**
   - Navigate to the "Math / Backend Config" tab.
   - Upload `game_config.json`. This securely binds the physical odds into Stake's proprietary Python/Go server environment so it can run the spins without trusting your frontend.
6. **Documentation Submission:**
   - Upload the `docs/GAME_ARCHITECTURE.md` file as supporting evidence. This proves to their compliance engineers that the game decouples RNG from the client.

### Step C: Staging & Certification
Once you press submit, the game moves to Stake's **Staging Server**.
- You will be provided a test link (e.g., `staging.stake.com/casino/sweet-cluster/demo`).
- Send this link directly to your client. 
- The client will run through their own compliance checks. Since we have enforced strict RGS decoupling and handled all edge cases, it should pass Certification upon their first initial review.
